What Powers the Pipeline

Four technologies work together to take your code from a laptop to a globally available service — automatically.

Docker

Packages your app + all its dependencies into a single container image. Think of it as a shipping container for software — runs identically on your laptop, in CI, and in the cloud.

Terraform

Runs your Docker containers without managing servers. You say "run this container with 512MB RAM" and AWS handles the rest — no EC2 instances, no patching, no SSH.

Automates your test → build → deploy pipeline. Every push triggers tests; every merge to main triggers deployment. Humans write code, machines do the rest.

Docker What is a Container?

A container is a lightweight, standalone package that includes your application code, runtime, libraries, and system tools. Unlike a virtual machine, containers share the host OS kernel, making them fast and efficient.

  • Dockerfile — recipe that describes how to build the image
  • Image — the built artifact (like a compiled binary, but for your whole app)
  • Container — a running instance of an image
  • docker-compose — runs multiple containers together (frontend + backend)
# Our backend Dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app"]

Terraform Infrastructure as Code

Instead of manually creating AWS resources via the console, Terraform lets you describe your infrastructure in .tf files. Run terraform apply and it creates, updates, or destroys resources to match your code.

  • Declarative — you describe the desired state, Terraform figures out the steps
  • State file — tracks what currently exists so it only changes what's different
  • Plan — preview changes before applying them (safe!)
  • Idempotent — running apply twice with no changes = no effect
# Create an ECS cluster in Terraform
resource "aws_ecs_cluster" "main" {
  name = "paper2notebook"
}

resource "aws_ecs_service" "backend" {
  cluster        = aws_ecs_cluster.main.id
  launch_type    = "FARGATE"
  desired_count  = 2
}

AWS ECS Fargate Serverless Containers

ECS (Elastic Container Service) is AWS's container orchestrator. Fargate is the serverless mode — you don't provision or manage any EC2 instances. Just define CPU/memory per container and AWS handles the rest.

  • Task Definition — blueprint for a container (image, CPU, memory, ports, env vars)
  • Service — keeps N copies of a task running, replaces crashed ones
  • Cluster — logical grouping of services & tasks
  • Fargate vs EC2 — Fargate = no servers; EC2 = you manage instances
# What happens under the hood
You: "Run my-image with 512MB RAM"
AWS: Allocates compute
AWS: Pulls image from ECR
AWS: Starts container in your VPC
AWS: Connects to load balancer
AWS: Streams logs to CloudWatch
You: Do nothing. It just works.

CI/CD Continuous Integration & Deployment

CI runs automated checks on every code change. CD automatically deploys code that passes all checks. Together, they ensure only tested, secure code reaches production.

  • CI: pytest — runs all backend unit + integration tests
  • CI: Playwright — runs browser-based E2E tests
  • CI: semgrep — scans code for security vulnerabilities
  • CI: pip-audit — checks dependencies for known CVEs
  • CD: auto-deploy — merge to main → build → push → deploy
# .github/workflows/ci.yml
on: [push, pull_request]
jobs:
  test: pytest --cov
  e2e: npx playwright test
  security: semgrep scan
  audit: pip-audit
  deploy: # only on main, after all pass
    docker build && push && ecs update

Docker Builds Container Images

Two Docker images are built: one for the FastAPI backend and one for the React frontend (served by Nginx). Each image is a self-contained, portable unit.

# Build both images
$ docker build -t paper2notebook-backend -f backend/Dockerfile .
$ docker build -t paper2notebook-frontend -f frontend/Dockerfile .

# Push to AWS ECR (private registry)
$ docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/backend
$ docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/frontend

Terraform Ensures Infrastructure Exists

Terraform compares the desired state (your .tf files) with the actual state (what's currently in AWS). It creates, updates, or destroys only what needs to change.

Plain English

How Everything Connects

The entire workflow, explained like you're telling a friend over coffee.

The Story of a Single Code Change

1

You write a bug fix on your laptop and run git push. Your code + the Dockerfiles + the Terraform configs all live in the same repo.

2

GitHub receives the push and immediately starts the CI pipeline. Think of it as a robot coworker that reviews every change you make. It runs four checks in parallel: pytest (does the code work?), Playwright (does the UI work?), semgrep (is the code secure?), and pip-audit (are the dependencies safe?).

3

If any check fails, the merge is blocked. No bad code reaches production. You fix the issue and push again.

4

Once all checks pass, the CD pipeline kicks in. It takes your code and uses the Dockerfiles to build two container images — one for FastAPI, one for React+Nginx. Think of these as two sealed boxes, each containing everything needed to run that service.

5

The images are pushed to AWS ECR (Elastic Container Registry) — your private Docker Hub inside AWS. This is where ECS will pull the images from.

6

If the infrastructure changed (new service, different CPU allocation, etc.), Terraform runs terraform apply. It reads your .tf files, compares them with what's currently in AWS, and makes only the necessary changes. If nothing changed, it does nothing.

7

ECS Fargate pulls the new images from ECR and starts fresh containers. Fargate is "serverless" — you never see or manage any servers. You just say "run this container with 512MB RAM and 0.25 vCPU" and AWS figures out where to put it.

8

The Application Load Balancer (ALB) sits in front of both containers. When a user visits your URL, the ALB looks at the path: /api/* requests go to the FastAPI backend, everything else goes to the React frontend. It also does health checks — if a container stops responding, the ALB stops sending traffic to it.

9

Meanwhile, CloudWatch collects every log line from both containers. If something breaks at 3 AM, you can see exactly what happened. IAM roles ensure each service has minimum permissions — the containers can pull images and write logs, nothing more.

✓

That's it. From git push to live users — fully automated, tested, secured, and monitored. You wrote code, pushed it, and the pipeline handled the rest. This is what "production-ready" means.
